K7820: Overview of SNAT features

Non-Diagnostic

Original Publication Date: Sep 22, 2015
Updated Date: Oct 23, 2020

Applies to (see versions):
  • Product: BIG-IP, BIG-IP AAM, BIG-IP AFM, BIG-IP APM, BIG-IP ASM, BIG-IP Link Controller, BIG-IP LTM, BIG-IP PEM

Overview

A Secure Network Address Translation (SNAT) is an object that maps the source client IP address in a request to a translation address defined on the BIG-IP device. When the BIG-IP system receives a request from a client, and if the client IP address in the request is defined in the origin address list for the SNAT, the BIG-IP system translates the source IP address of the incoming packet to the SNAT address.

A SNAT can be used by itself to pass traffic that is not destined for a virtual server. For example, you can use a SNAT object to pass certain traffic (such as DNS requests) from an internal network to an external network where your DNS server resides.

A SNAT can also be used in conjunction with a virtual server and destination address of a NAT (displayed as NAT address in the Configuration utility) to translate the source IP address of an incoming connection. (With no SNAT configured, no source address translation takes place). In this case, the SNAT objects match traffic after the traffic has already matched a virtual server or NAT and, therefore, translates the source address upon egress to the server side, unless that traffic is already subjected to SNATs that are applied to the virtual server.

For example, when the BIG-IP system receives a new connection from source IP address 192.168.20.1 to destination IP address 192.168.10.1, the virtual server or NAT listener on 192.168.10.1 accepts the connection, and upon egress to the server side, the SNAT listener 192.168.20.0/24 processes the connection and translates the source address to the Translation address that is configured on the SNAT object. If there is a source address translation configured on the virtual server, this address is used instead rather than the SNAT object. Continuing with the example, the system translates 192.168.20.1 to 172.16.20.1. As a result of this behavior, you can configure a single SNAT object to translate traffic from all virtual servers destined to the next hop on the server side, as opposed to configuring the Source Address Translation property on every virtual server. In this scenario, when you want to bypass the SNAT for a specific virtual server (which has a pool configured) and allow the client address on the server side, you can disable the associated pool setting Allow SNAT.

You can also use a SNAT to ensure that response traffic is returned through the BIG-IP system without requiring other outbound non-load balanced traffic to also route through the BIG-IP system, and without requiring any changes in the router or server's configuration. SNAT is also a critical component in one-armed configurations, preventing the server from responding directly to the client.

A SNAT works in the following manner:

  1. The BIG-IP system receives a request directly from a client or from virtual server traffic and verifies whether that source IP address is defined in the origin address list for the SNAT.
  2. If the client's IP address is defined in the origin address list for the SNAT, the BIG-IP system translates the source IP address to the translation address defined in the SNAT.
  3. The BIG-IP system then sends the client request to the pool member or other destination.

Types of SNATs

Standard SNATs and intelligent SNATs are illustrated in the following sections:

Standard SNATs

The following three examples illustrate three types of standard SNATs:

A SNAT in which you specify a specific translation address

One way to create a SNAT is to directly map one or more original IP address to a specific translation address that you choose. For the SNAT origin address, you can specify host addresses, network addresses, or a wildcard that matches all addresses. For example, the following tmsh command translates the address of connections that originate from the address 10.10.10.1 to the translation address 172.16.0.1:

create /ltm snat /Common/test_snat origins add { 10.10.10.1/32 }   translation 172.16.0.1

Note: Ensure that you specify the route domain ID on the origin and translation address when translating addresses for specific route domains. For example: 10.10.10.1%5/32.

Note: For more information about SNATs and SNAT pools, refer to K47945399: Create basic SNATs and SNAT pools.

Automap SNAT

Of the available SNAT options, SNAT automap is often preferred because it is simple to configure and maintain, and helps conserve IP addresses by using the BIG-IP system's existing self IP addresses.

When the BIG-IP system processes connections from the origin IP addresses matching a SNAT automap definition, it chooses a translation address from the available self IP addresses. Floating self IP addresses on the egress VLAN are preferred to support seamless failover. If multiple floating self IP addresses are configured on the VLAN, the BIG-IP system translates the address of client connections by alternating through a pool of all floating self IPs on the VLAN.

Note: The SNAT automap feature may not use the intended translation address if a floating self IP is not available on the egress VLAN, or the floating self IP address was originally a static self IP address. For more information, refer to K7336: The SNAT Automap and self IP address selection.

For example, the following tmsh command translates the address of connections that originate from the address 10.10.10.1 to one of the system's self IP addresses:

create /ltm snat /Common/test_snat automap origins add { 10.10.10.1/32 }

The following tmsh command translates addresses of the network range 10.10.0.0/16:

create /ltm snat /Common/test_snat automap origins add { 10.10.0.0/16 }

SNAT pools

A SNAT pool represents a pool of translation addresses that you configure on the BIG-IP system. The original IP address is then mapped to the entire translation pool, called a SNAT pool. For example, the following tmsh command contains the translation addresses 172.16.0.1 and 172.16.0.2:

create /ltm snatpool /Common/my_snatpool members add { 172.16.0.1 172.16.0.2 }

After you create the SNAT pool, you must associate it with a SNAT object. For example, the following tmsh command translates the address of connections that originate from the address 10.10.10.1 to one of the IP addresses in the SNAT pool:

create /ltm snat /Common/test_snatpool origins add { 10.10.10.1/32 { } } snatpool my_snatpool

Important: When using a SNAT pool with IP addresses from the egress VLAN (the VLAN for which the packet exits in the BIG-IP system) and non-egress VLAN networks, the egress VLAN network address is given higher priority. For example, egress VLAN external has a self IP of 172.16.0.254/24, and SNAT pool member addresses of 172.16.0.1/24 and 10.1.1.1/24. The BIG-IP system prefers the egress VLAN SNAT pool member address 172.16.0.1 and continues to use the same address until it becomes unavailable.

Note: The BIG-IP system load balances SNAT pool connections between members using the least connections algorithm.

Note: For more information about SNATs and SNAT pools, refer to K47945399: Create basic SNATs and SNAT pools.

Intelligent SNATs

An intelligent SNAT is the mapping of one or more original client IP address to a translation address. However, you implement this type of SNAT mapping within an iRule. An intelligent SNAT allows the BIG-IP system to base its selection of a translation address on any piece of packet data that you specify. This piece of data could be the original client IP address, or it could be another piece of data in the packet, such as a server port or an HTTP cookie.

To configure an intelligent SNAT, you must complete the following tasks:

  • Determine the type of packet data that the BIG-IP system uses as a basis for selecting a translation address, such as the server port.
  • Create the SNAT or SNAT pools that the BIG-IP system uses to select a translation address.
  • Assign the iRule as a resource to the virtual server.

The following two examples illustrate mapping original client IP addresses to a translation address using an iRule:

Example 1

If you want the BIG-IP system to base its selection of a translation address on the destination port, you would first create a data group that contains the destination ports, and then create the iRule that applies the SNAT translation address to connections using a port specified in the data group. After you have created the data group and SNAT, you must assign the iRule as a resource to the virtual server. The following tmsh command creates a data group called Ports, containing ports 80, 81, and 8080:

create /ltm data-group internal Ports type string records add { 80 81 8080 }

After you create the data group, create the iRule that applies the SNAT translation address to connections using ports from the Ports data group. The following iRule examples apply the SNAT translation address of 172.16.0.1 to connections using ports from the Ports data group:

  1. Create the iRule using tmsh by entering the following command:

    create /ltm rule Ports_Snat_iRule

  2. Place a stanza similar to the following between the opening and closing braces:

    when CLIENT_ACCEPTED {
    if { [class match [TCP::local_port] equals Ports]} {
    snat 172.16.0.1
    }
    }

  3. After the system confirms the changes, press ESC to quit interactive mode, then enter :wq.
  4. Select y to save changes.
  5. Apply the iRule to the virtual server using the following command syntax:

    modify /ltm virtual ftp_vs rules { Port_Snat_iRule }

  6. Save the configuration by entering the following command:

    save /sys config partitions all

Example 2

If you want the BIG-IP system to base its selection of a translation address on the client/source IP address and the destination port, and then forward unchanged traffic that does not match this criteria, you would first create two data groups that contain the client/source IP addresses and destination ports respectively, and then create the iRule that would apply the SNAT translation address.

The following tmsh command creates a data group called Hosts, which contains IP addresses 10.10.10.1, 10.10.10.2, and 10.10.10.3:

create /ltm data-group Hosts type ip records add { 10.10.10.1 10.10.10.2 10.10.10.3 }

The following tmsh command creates a data group called Ports, which contains ports 80 and 8080:

create /ltm data-group Ports type string records add { 80 8080 }

After you create the data groups, create the iRule that applies the SNAT translation address to connections using IP addresses and ports from the Hosts and Ports data groups, and forward all other connections. The following iRule example apply the SNAT translation address of 172.16.0.1 to connections using IP addresses and ports from the Hosts and Ports data groups, and forward all other connections:

  1. To create the iRule using tmsh, enter the following command:

    create /ltm rule Hosts_Ports_Snat_iRule

  2. Place a stanza similar to the following between the opening and closing braces:

    when CLIENT_ACCEPTED {

    if { [class match [IP::client_addr] equals Hosts]} {
    if { [class match [TCP::local_port] equals Ports]} {
    snat 172.16.0.1
    } else {
    forward
    }
    }
    }

  3. After the system confirms the changes, press ESC to quit interactive mode, then enter :wq.
  4. Select y to save changes.
  5. Apply the iRule to the virtual server using the following command syntax:

    modify /ltm virtual ftp_vs rules { Hosts_Ports_Snat_iRule }

  6. Save the configuration by entering the following command:

    save /sys config partitions all

SNAT port exhaustion

Port exhaustion or collisions may occur under heavy usage or special client traffic patterns. As a result, connections that cannot be translated due to lack of available ports on a given translation address may be dropped.

When a SNAT is configured on the BIG-IP system (either by itself or in conjunction with a virtual server), the source address of each connection is translated to a configured SNAT address, and the source port is mapped to a port currently available for that address. By default, the BIG-IP system attempts to preserve the source port, but if the port is already in use on the selected translation address, the system also translates the source port.

Note: Starting in BIG-IP 10.0.0, it is possible to control, on a per-virtual-server basis, whether the system should preserve the client's source port. For more information, refer to K11003: Configuring source port preservation for virtual servers.

Each SNAT address, like any IP address, has only 65535 ports available. This is a limit of the TCP and User Datagram Protocol (UDP) protocols, since they use a 16-bit unsigned integer (thus ranging from 0 to 65535) to specify the source and destination ports. However, each SNAT address can potentially process more than 65535 concurrent connections, as long as each socket pair is unique. A socket pair is defined by a 4-tuple structure consisting of the following elements:

  • Source IP address
  • Source port
  • Destination IP address
  • Destination port

For example, a given SNAT address can continue to use the same source port as long as the remote socket is unique, thus allowing the SNAT address to process more than 65535 concurrent connections.

For example:

SNAT address and port Remote socket
10.1.1.1:1234 -------------> 10.1.1.200:80
10.1.1.1:1234 -------------> 10.1.1.201:80
10.1.1.1:1234 -------------> 10.1.1.200:8080
10.1.1.1:1234 -------------> 10.1.1.201:8080

Note: When SNAT is used in conjunction with a virtual server that load balances connections to a pool, the remote socket is the IP address and port of the chosen pool member. Therefore, assuming a certain SNAT address is configured on only one virtual server, the SNAT address is able to process approximately 65535 concurrent connections for each pool member in the pool (each unique remote socket).

While the uniqueness of remote sockets depends entirely on your specific configuration and traffic, for simplicity you should think of 65535 concurrent connections as the maximum capacity for any given SNAT address. If you think more than 65535 connections may require translation, you should configure more SNAT addresses (for example, using a SNAT pool).

You may be able to determine when SNAT port exhaustion is occurring by reviewing the system log files. When port exhaustion occurs, error messages that appear similar to the following examples are logged to the /var/log/ltm file:

01010201:2: Inet port exhaustion on 10.1.21.26 to 172.28.21.71:53 (proto 17)
01010201:2: Inet port exhaustion on 10.10.10.211 to 172.28.21.123:80 (proto 6)

Note: The previous error messages are not specific to SNAT port exhaustion and may be logged whenever the Traffic Management Microkernel (TMM) detects port exhaustion for a traffic management object.

Beginning in BIG-IP 12.0.0, the system issues an early warning when it detects that the pool of available service ports is nearing exhaustion. For more information about configuring the port exhaustion message threshold, refer to K63275550: Modifying the BIG-IP system's ephemeral port exhaustion warning.

You can monitor the number of concurrent connections going through the SNAT by running the tmsh show /ltm snat command, or in the Configuration utility, choose one of the following methods:

BIG-IP 12.0.0 and later

  1. Log in to the Configuration utility.
  2. Go to Statistics > Module Statistics > Local Traffic.
  3. For Statistics Type, select SNATs, SNAT Pools, or SNAT Translations.

BIG-IP 11.0 and earlier

  • Go to Overview > Statistics > Local Traffic > Statistics Type and select SNATs.

For information about mitigating SNAT port exhaustion, refer to Using SNAT pools or SNAT automap to prevent port collisions in the SNAT uses and the following best practices section.

SNAT uses and best practices

When planning for SNAT traffic, you must consider the following factors:

  • Number of concurrent connections
  • Connection reuse
  • TCP/UDP idle timeout

To maximize the availability of the ephemeral SNAT ports, limit the amount of time that a TCP/UDP port is held in an idle state by reducing the SNAT timeout value to the smallest interval possible for the traffic that the SNAT supports. For example, HTTP traffic does not require a long timeout since HTTP uses short-lived, active connections. A 60-second TCP idle timeout is more than ample for general HTTP traffic. If the SNAT supports FTP traffic, the timeout value must take into account the data flowing over both the control and data channels: the control channel may go idle while data is being sent on the data channel, so the SNAT timeout should reflect the transfer time of the longest expected data download. In general, a 120 to 300-second TCP idle timeout is more than sufficient for FTP traffic.

Judicious use

Enable SNAT only where necessary, to preserve available source IP addresses and ports.

Allowing internal hosts to access external devices through the BIG-IP system

If you have internal hosts that are configured to route to the BIG-IP system, and the hosts require access external devices through the BIG-IP system, you can create a SNAT. For example, you can create a SNAT object on the BIG-IP system to pass certain administrative traffic such as DNS or SNMP from an internal network to an external network. The SNAT translates the source IP address of the packet to the SNAT address so that the external device routes the reply packet back to the BIG-IP system.

Connecting to a virtual server with a pool member that is on the same IP subnet as the client

If you want to load balance requests to nodes that are on the same network as the clients, you must configure a SNAT on the BIG-IP system so that connections and responses are passed through the BIG-IP system. This is commonly referred to as a one-armed configuration.

Using SNAT pools or SNAT automap to prevent port collisions

SNATs have a limit of 65535 ports. SNAT connections can fail if a large number of client's requests are traversing the SNAT. To mitigate port collisions, create SNAT pools or use SNAT automap with an appropriate number of self IP addresses on the VLAN to support the expected level of concurrent connections using SNAT.

SNAT demotes PVA acceleration

A SNAT demotes Packet Velocity ASIC (PVA) acceleration on systems that are equipped with a PVA chip. When a SNAT is associated with a virtual server, the virtual server is demoted to partial PVA acceleration.

SNATs do not preserve the client address

When the BIG-IP system translates the source IP address of the incoming packet to the SNAT address, the web server sees the request as originating from the SNAT address, not the original client IP address. If the web server is required to log connections using the original client IP address, SNAT address translation prevents this.

SNATs only forward TCP and UDP traffic

By default, SNATs forward only TCP and UDP traffic. Other IP traffic, such as Internet Control Message Protocol (ICMP), is dropped. You can configure SNATs to forward any IP packets by modifying the SNAT Packet Forwarding setting. For information, refer to K3760: Configuring SNATs to forward any IP protocols.

SNAT automap and redundant BIG-IP systems

If you decide to use SNAT automap on a BIG-IP high availability (HA) pair, ensure that at least one floating self IP address exists on each egress VLAN. This ensures traffic translates to the same translation address (the floating self IP address) regardless of which unit is currently active.

SNAT automap idle timeout

SNAT automap objects have a non-configurable idle timeout value. If you need to implement a SNAT with a configurable idle timeout, create a SNAT with a defined translation IP address or a SNAT pool, and then set the desired idle timeout for the translation addresses. You can specify a timeout value in seconds or use the keyword Indefinite. For more information, refer to K6017: The BIG-IP SNAT automap idle timeout value is not configurable.
